## 🟡 中等 Bug（功能不完整）

### Bug #4: `commissionService` 调用时机不完整

**影响范围：** P1-3 佣金结算

**根因：** `order.js` 的 `POST /orders/:id/delivery`（交付签收）已正确调用 `commissionService.autoGenerate(id)` ✅，但 `PUT /orders/:id/installation/confirm`（安装确认）**没有**调用。

**计划逻辑：** 按设计，佣金应在"安装完成确认"时生成（`installation/confirm`），而不仅是交付签收时。

**修复：** 在 `PUT /orders/:id/installation/confirm` 的成功响应前添加：
```javascript
const dealerId = order.rows[0].dealer_id;
if (dealerId) {
  require('../services/commissionService').autoGenerate(id)
    .catch(err => logger.error('佣金自动生成失败:', err));
}
```

### Bug #5: `dealer.js` 缺少 `webhookService` 顶层引用

**影响范围：** P1-2 Webhook

**根因：** `dealer.js` 的 Webhook 配置路由（查询/创建/删除 Webhook）没有 `require('../services/webhookService')`，Webhooks 只能通过 `order.js`/`shipment.js`/`finance.js` 触发，但无法从 dealer.js 的经销商操作中触发。

**说明：** dealer.js 的 Webhook CRUD 功能本身可以正常运行（直接操作数据库），只是缺少顶层引用使代码不规范。

**修复：** 在 `dealer.js` 顶部添加：
```javascript
const webhookService = require('../services/webhookService');
```

## 🔵 轻微 Bug（代码质量/可维护性）

### Bug #6: `commissionService.autoGenerate` 错误处理位置问题

**根因：** `autoGenerate` 的 Webhook 触发在 `try` 块外部（line 67-73），如果 Webhook 抛出异常不会被捕获，但外层 `catch` 在 line ~80 会捕获。

**影响：** 实际影响极小（WebHook 失败不影响佣金生成），但代码结构不清晰。

---

### Bug #7: `production.js` 的 `/voice` 路由是占位实现

**根因：** `POST /api/production/voice`（line 616-623）只返回成功响应，没有实际调用语音服务。真正的语音服务在 `POST /api/hardware/voice`。

**影响：** 前端调用 `/api/production/voice` 无实际效果，但不影响生产（因为有 hardware 接口可用）。

**修复：** 将 `production/voice` 改为转发给 `hardware/voice`。

---

### Bug #8: `schedulingService` 的 `enqueue` 是后加的未充分测试

**根因：** `enqueue` 方法是本轮新增的（修复 Bug），但还未在实际流程中充分测试。`order.js` 已添加 `enqueue` 调用，但需要数据库有 `production_schedule` 表（migration 007）和后端重启后才生效。
