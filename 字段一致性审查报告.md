# 橱柜工厂管理系统 - 字段一致性审查报告

审查日期：2026-04-25
审查范围：数据库层（SQL）、后端路由层、前端API层

---

## 一、字段命名问题清单（按模块分组）

### 1.1 时间戳字段混乱 P0

| # | 问题描述 | 涉及表/文件 | 建议统一写法 |
|---|---------|-----------|------------|
| T1 | 部分表用 created_at，部分用 create_at，部分用 create_time | init_database.sql 中所有45+表；add_missing_tables.sql | 统一为 created_at |
| T2 | order_master 用 create_time，但 JOIN 的子表用 created_at | init_database.sql | 统一为 created_at |
| T3 | add_missing_tables.sql 中 dealer_customer_follow、dealer_application，dealer_customer 表使用 create_time 而非 created_at | add_missing_tables.sql | 批量改为 created_at |
| T4 | production_schedule 表使用 scheduled_date、actual_start_date、actual_end_date，与其他表的时间字段风格不一致 | init_database.sql | 改为 scheduled_at/actual_started_at/actual_ended_at |

根因：建表 SQL 早期使用了不同的命名约定，后期迁移脚本未完全统一。

### 1.2 金额字段混乱 P0

| # | 问题描述 | 涉及表/文件 | 建议统一写法 |
|---|---------|-----------|------------|
| A1 | 应收款模块：receivable 表（旧）用 amount，而 accounts_receivable 表（新）用 total_amount | init_database.sql、finance.js、receivable.js | 统一为 total_amount，废弃旧 receivable 表 |
| A2 | supplier_payment 表用 amount，payable 表用 total_amount | init_database.sql | 统一为 total_amount |
| A3 | fund_flow 表用 amount，与 receivable/payable 的 total_amount 不一致 | init_database.sql | 统一为 total_amount |
| A4 | order_detail 用 detail_amount（带前缀），order_master 用 total_amount | init_database.sql | 明细金额直接用 amount 即可 |
| A5 | stock_in 用 total_amount，stock_in_detail 用 total_price | init_database.sql | 统一为 total_amount |
| A6 | collection_record 用 collection_amount，payment_record 用 payment_amount | init_database.sql | 建议统一为 amount |
| A7 | 发货单 shipment 表无 total_amount 字段，仅有 shipped_boards 计数 | init_database.sql、shipment.js | 建议补充 total_amount 字段 |

### 1.3 客户/经销商字段名不一致 P0

| # | 问题描述 | 涉及表/文件 | 建议统一写法 |
|---|---------|-----------|------------|
| C1 | order_master 既有 customer_id 又有 customer_name（冗余），同时还有 dealer_id | init_database.sql | 去掉 customer_name 冗余字段，通过 JOIN 获取 |
| C2 | accounts_receivable 同时有 customer_id、customer_name（冗余）、dealer_id 三个关联字段 | init_database.sql | 去掉冗余 customer_name，通过 JOIN 获取 |
| C3 | receivable.js 路由中，GET / 查 receivable 表（按 dealer_id），GET /list 查 accounts_receivable 表（按 customer_id），两套数据结构完全不一致 | receivable.js | 合并为一套，统一用 accounts_receivable 表 |

### 1.4 状态枚举值不统一 P0

| # | 问题描述 | 涉及表/文件 | 建议统一写法 |
|---|---------|-----------|------------|
| S1 | receivable 表的 status（paid/unpaid）与 accounts_receivable.payment_status（paid/unpaid/partial）字段名和枚举值都不同 | init_database.sql | 统一为 payment_status，枚举为 paid/partial/unpaid |
| S2 | cabinet_board.current_location 存储中文文本值如已发货、已出库、入库区，无法建立 CHECK 约束，SQL 字符串比较有隐患 | init_database.sql、warehouse.js、shipment.js | 改为 location_status VARCHAR(20) 英文枚举：in_storage/shipped/out_of_stock |
| S3 | warehouse_location.status 用 empty/occupied（英文），但 cabinet_board.current_location 用中文，命名相似但语言不统一 | init_database.sql | 统一为英文枚举 |

### 1.8 后端路由层两套数据模型冲突 P0（严重）

| # | 问题描述 | 涉及文件 | 风险 |
|---|---------|---------|------|
| B1 | receivable.js 中 GET / 与 GET /list 查询不同表（receivable vs accounts_receivable），且 finance.js 的 GET /finance/receivables 也查 receivable 表，同一业务三套查询逻辑 | receivable.js、finance.js | 高：数据源不统一，前端处理复杂 |
| B2 | supplier.js 的 POST /:id/payments 和 finance.js 的 POST /payables/:id/pay 都写 fund_flow 表，可能产生重复流水 | supplier.js、finance.js | 高：资金流水重复记录 |
| B3 | production.js 的 PUT /schedule/:scheduleId 接收 worker_id/worker_name/scheduled_date，但 production_schedule 表的实际字段名需确认一致性 | production.js、init_database.sql | 中：字段名不匹配可能导致更新失败 |

### 1.9 前端API层不一致 P1

| # | 问题描述 | 涉及文件 | 建议统一写法 |

| F2 | 前端 warehouse.finishedLocations 调用 /warehouse/finished-locations（kebab-case），locations 调用 /warehouse/locations | index.js、warehouse.js | 统一路由命名风格（推荐 kebab-case） |
| F3 | 前端 shipment.list 调用 /shipment/（尾部斜杠），orders 调用 /shipment/orders | index.js、shipment.js | 统一去掉尾部斜杠 |

## 三、推荐统一规范

### 3.1 时间戳字段规范
- 创建时间：统一 created_at（TIMESTAMP NOT NULL DEFAULT NOW()）
- 更新时间：统一 updated_at（TIMESTAMP NOT NULL DEFAULT NOW()）
- 业务日期：统一 {entity}_date（如 bill_date、payment_date）
- 计划/实际时间：统一 {entity}_at（如 scheduled_at、actual_started_at）
- 禁止使用：create_time、update_time、create_at、update_at

### 3.2 金额字段规范
- 总金额：统一 total_amount（DECIMAL(14,2)）
- 已付/已收：paid_amount（DECIMAL(14,2)）
- 余额：balance_amount（DECIMAL(14,2)，NOT NULL DEFAULT 0）
- 单价：统一 unit_price（DECIMAL(14,2)）
- 折扣后最终金额：final_amount（DECIMAL(14,2)）
- 币种：currency（VARCHAR(10)，如 CNY/USD）
- 人民币折算：{field}_cny（DECIMAL(14,2)）
- 禁止使用：amount、money、total、cost（语义模糊）

### 3.3 外键字段规范
- 命名：统一 {entity}_id（全小写，UUID 类型）
- 禁止使用：字符串类型的 order_no 作为外键
- 主键：统一使用 UUID v4（gen_random_uuid()）

### 3.4 状态字段规范
- 订单状态：order_status（VARCHAR(20)，英文枚举）
- 支付状态：统一 payment_status（VARCHAR(20)，英文枚举：paid/partial/unpaid）
- 通用状态：status（VARCHAR(20)，英文枚举）
- 板件位置：改为 location_status VARCHAR(20) 英文枚举
- 禁止使用：中文文本作为枚举值

### 3.5 联系方式字段规范
- 手机/电话：统一 phone（VARCHAR(20)）
- 邮箱：统一 email（VARCHAR(100)）
- 联系人姓名：统一 contact_name（VARCHAR(100)）
- 禁止混用：tel、mobile、telephone

### 3.6 编号字段规范
- 命名格式：{entity}_no（如 order_no、quote_no、bill_no）
- 编号前缀：订单 ORD / 报价 Q / 收据 CR / 付款 PA / 入库 IN / 出库 OUT

### 3.7 NULL 约束规范
- 外键：NOT NULL（级联删除在应用层处理）
- 金额字段：NOT NULL DEFAULT 0
- 时间戳字段：NOT NULL DEFAULT NOW()
- 可选文本字段（地址、备注）：NULL DEFAULT NULL

## 六、审查结论

### 系统整体评估

该系统存在三类主要问题：

**1. 历史遗留的命名不统一**
早期建表 SQL 使用了不同的命名约定（create_time vs created_at、金额字段混用 amount/total_amount），后期迁移脚本未完全统一。

**2. 两套数据模型并存（最严重）**
应收款模块存在 receivable（旧）和 accounts_receivable（新）两套表，后端路由同时读写两套表，前端需要分别处理，增加复杂度并引入数据不一致风险。

**3. 板件位置使用中文文本枚举**
cabinet_board.current_location 直接存储中文文本（如已发货），会导致：
- SQL 字符串比较受空格、全角/半角影响
- 无法建立 CHECK 约束
- 排序和统计困难
- 前端需要硬编码中文字符串

### 修复优先级建议

**第一阶段（P0，阻塞性问题）：**
- 统一时间戳字段命名（create_time -> created_at）
- 统一金额字段命名（amount -> total_amount）
- 解决应收款两套数据模型问题
- 将板件位置改为英文枚举

**第二阶段（P1，可维护性提升）：**
- 补充缺失的编号字段
- 统一联系方式字段命名
- 统一报价模块联系人字段
- 前端 API 端点统一
